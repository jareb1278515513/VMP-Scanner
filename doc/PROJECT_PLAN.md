# 项目计划

## 1 项目名称

VMP-Scanner: Visible, Measurable, and Perceivable Network Scanning and Vulnerability Detection System

VMP-Scanner：“可见、可测、可感”的网络扫描与漏洞检测系统

## 2 项目背景与目标

### 2.1 项目背景

在教学、靶场和内部安全自查场景中，常见问题是“能扫到资产，但无法快速定位风险优先级”。本项目以 DVWA 作为标准化训练环境，构建一个从资产发现、漏洞探测到风险量化的自动化工具链。

### 2.2 项目总目标

实现一个基于 Python 的分层安全审计系统，满足以下能力：

1. 可见：完整发现网络资产、Web 路由与输入面。
2. 可测：通过插件化漏洞脚本执行可重复测试。
3. 可感：将技术结果映射为可理解、可排序、可追踪的风险结论。

### 2.3 阶段性目标

1. M1（基础可用）：完成采集层，输出标准化资产清单。
2. M2（能力扩展）：实现 3 至 5 类基础漏洞检测插件并形成统一执行框架。
3. M3（结果闭环）：完成风险评分与报告展示，支持对比与复测。

## 3 项目范围

### 3.1 本期范围（In Scope）

1. 目标环境：DVWA（本地或内网实验环境）。
2. 资产发现：端口扫描、URL 枚举、表单与参数收集。
3. 漏洞检测：SQL 注入、XSS、弱口令/暴力尝试、敏感路径探测、CSRF 基础检测（按插件逐步实现）。
4. 风险评估：规则评分 + 置信度 + 修复建议。
5. 表现层：命令行输出 + 结构化报告（JSON/Markdown）。

### 3.2 非本期范围（Out of Scope）

1. 大规模分布式扫描。
2. 真实互联网未授权目标扫描。
3. 自动化利用链与后渗透能力。

## 4 环境与依赖管理

### 4.1 技术栈

1. 语言：Python 3.11+
2. 环境管理：uv
3. 网络与 HTTP：socket、requests
4. 解析：BeautifulSoup4
5. 输出与日志：rich（可选）、logging、json
6. 测试：pytest

### 4.2 环境规范

1. 使用 `uv` 统一创建虚拟环境和锁定依赖。
2. 固定依赖版本，避免“同样代码不同结果”。
3. 开发、测试、演示环境使用一致的 Python 小版本。

### 4.3 建议命令约定

1. 初始化：`uv sync`
2. 运行：`uv run python main.py`
3. 测试：`uv run pytest`

## 5 分层架构设计

系统采用四层架构：采集层 -> 漏洞探测层 -> 风险评估层 -> 表现层。所有层之间通过统一数据模型解耦，保证可扩展性。

### 5.1 采集层（Collection Layer）

#### 5.1.1 引擎 A：原生网络探测引擎（Network Discovery Engine）

1. 职责：识别主机开放端口与基础服务指纹，确定网络攻击面。
2. 输入：目标 IP/网段、端口范围、超时和并发参数。
3. 核心实现：
	1. 基于 socket 的 TCP connect 扫描。
	2. 端口状态判定（open/closed/filtered）。
	3. 可选 banner 抓取（轻量、低侵入）。
4. 输出：`host`, `port`, `status`, `service_guess`, `response_time_ms`。
5. 性能目标：在实验环境中可配置并发，避免对目标造成明显负载冲击。

#### 5.1.2 引擎 B：Web 状态感知爬虫引擎（Web Crawler Engine）

1. 职责：发现应用入口、参数面与可疑接口，形成应用层攻击面。
2. 输入：起始 URL、最大深度、域名白名单、会话 Cookie。
3. 核心实现：
	1. 页面抓取与链接提取（a/form/script）。
	2. 去重策略（URL 标准化、参数排序、哈希去重）。
	3. 状态感知（登录态、重定向链、状态码统计）。
4. 输出：
	1. URL 清单（方法、参数、来源页、深度）。
	2. 表单清单（action、method、input 字段、token 字段）。
	3. 可疑接口清单（隐藏路径、返回特征、错误模式）。

### 5.2 漏洞探测层（Detection Layer）

#### 5.2.1 设计原则

1. 插件解耦：每类漏洞一个插件，统一生命周期接口。
2. 模式分离：测试模式（低风险验证）与攻击模式（高覆盖验证）严格区分。
3. 可审计：每次探测保留请求与证据片段，便于复核。

#### 5.2.2 插件接口建议

每个插件至少实现：

1. `metadata()`：漏洞名称、风险等级、前置条件。
2. `match(target)`：判断目标是否适配插件执行。
3. `probe(target, mode)`：执行检测逻辑。
4. `verify(result)`：减少误报的二次验证。
5. `evidence()`：输出关键证据（请求、响应摘要、payload）。

#### 5.2.3 初始插件清单

1. SQL 注入（参数注入 + 回显特征/错误特征）。
2. XSS（反射型输入回显检测）。
3. 弱口令/暴力尝试（登录入口策略评估，受频率限制）。
4. 敏感路径探测（常见管理路径/备份文件）。
5. CSRF 基础检测（关键表单 token 校验缺失识别）。

#### 5.2.4 Payload 字典来源与治理

1. 参考来源：以 PayloadsAllTheThings 作为漏洞 Payload 字典的主要参考来源。
2. 本地化策略：不直接全量使用，按插件类型筛选并沉淀为本地字典，避免无关或高破坏性 payload。
3. 分类结构：按漏洞类型维护字典（如 `sqli`, `xss`, `csrf`, `path_traversal`），每类区分测试模式与攻击模式。
4. 质量控制：每个 payload 记录适用场景、预期回显特征、风险等级和误报风险说明。
5. 安全边界：默认仅启用低风险验证 payload，高风险 payload 需显式开关并记录审计日志。
6. 版本管理：定期同步参考源更新，采用版本号与变更记录，确保结果可复现。

### 5.3 风险评估层（Assessment Layer）

#### 5.3.1 评估目标

将“扫描结果”转化为“可决策风险”，并给出修复优先级。

#### 5.3.2 评分模型（可迭代）

建议综合分数：

`RiskScore = Impact * Likelihood * Confidence * ExposureWeight`

1. Impact：业务与安全影响等级（1 至 5）。
2. Likelihood：被利用可能性（1 至 5）。
3. Confidence：证据可靠度（0.5 至 1.0）。
4. ExposureWeight：暴露面权重（内网、外网、认证前后）。

#### 5.3.3 输出内容

1. 风险清单（按分数降序）。
2. 风险分级（Critical/High/Medium/Low）。
3. 修复建议（短期缓解 + 长期治理）。
4. 复测建议（如何验证修复生效）。

### 5.4 表现层（Presentation Layer）

#### 5.4.1 输出渠道

1. CLI 实时输出：扫描进度、关键告警、统计摘要。
2. JSON 报告：供二次开发、可视化平台接入。
3. Markdown 报告：用于汇报、归档和复盘。

#### 5.4.2 报告结构建议

1. 基本信息：目标、时间、模式、工具版本。
2. 资产结果：开放端口、URL 与输入面统计。
3. 漏洞结果：漏洞项、证据、风险分、建议。
4. 附录：错误日志、跳过项、限制说明。

## 6 数据模型与流程

### 6.1 统一数据对象

1. Asset：主机、端口、服务、URL、表单。
2. Finding：漏洞发现项（类型、位置、证据、置信度）。
3. RiskItem：评估后的风险项（分数、分级、建议）。
4. Report：任务级输出（汇总、详情、元数据）。

### 6.2 处理流程

1. 输入目标与参数。
2. 采集层生成资产图。
3. 探测层根据资产分发插件并收集发现。
4. 评估层计算分值并排序。
5. 表现层输出报告与控制台摘要。

## 7 目录与模块规划

建议目录（可按实际代码调整）：

1. `main.py`：程序入口与任务编排。
2. `scanner/collection/`：采集层（资产发现）。
3. `scanner/collection/network/`：网络探测引擎实现。
4. `scanner/collection/crawler/`：Web 状态感知爬虫实现。
5. `scanner/detection/`：漏洞探测层（插件调度与执行）。
6. `scanner/detection/plugins/`：漏洞插件集合。
7. `scanner/detection/payloads/`：分漏洞类型维护 payload 字典与元数据。
8. `scanner/assessment/`：风险评估层（评分与分级）。
9. `scanner/presentation/`：表现层（CLI 与报告输出）。
10. `scanner/presentation/reporting/`：JSON/Markdown 报告渲染。
11. `tests/`：单元测试与集成测试。
12. `doc/`：设计文档、样例报告与实验记录。

## 8 开发计划与里程碑

### 8.1 里程碑定义

1. M1 采集层完成：
	1. 端口扫描可用。
	2. 爬虫可输出 URL/表单清单。
2. M2 探测层完成：
	1. 插件框架可加载与执行。
	2. 至少 3 个插件稳定运行。
3. M3 评估与表现层完成：
	1. 风险评分与分级落地。
	2. JSON 与 Markdown 报告可用。
4. M4 稳定性完善：
	1. 错误处理、日志、测试覆盖率提升。
	2. 文档与演示流程完善。
